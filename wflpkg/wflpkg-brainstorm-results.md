# WFL Package Manager: Brainstorm Results

## Problem Statement

WFL (Web First Language) is a natural-language programming language in which code reads like English. It ships an existing package manager, **wflpkg** (~6,000 lines of Rust plus a design document and a registry PRD for wflhub.org), but nothing is deployed and the project owner has explicitly said the current implementation "can be trashed and rebuilt if needed." The stated ambition is a package management experience on par with Cargo and npm in quality and safety. The question before the panel: is wflpkg's current design the best available, or can WFL do materially better before committing to code and a live registry?

wflpkg makes several distinctive bets that break sharply from mainstream ecosystems. The manifest (`project.wfl`) and lockfile (`project.lock`) are written in **prose natural-language syntax** with no TOML, JSON, quotes, or brackets (`requires http-client 26.1 or newer`). Versioning is **calendar-based (YY.MM.BUILD), not SemVer**, with constraints spelled in English ("26.1 or newer", "between 25.12 and 26.2"). Resolution selects a **single version per package**. There are **no install scripts** (pure source packages), **permission declarations** (`needs file-access`) confirmed at install time, and a planned registry (wflhub.org) with SHA-256/WFLHASH checksums, yank, typosquatting detection, OAuth, argon2, JWT, and a **flat namespace**. The design's rationale is that every choice traces to WFL's 19 foundational principles — chiefly natural-language syntax and "avoid unnecessary conventions" — and to the **No-Unlearning Invariant**, which requires that a beginner's form and an expert's form be the same form or a smooth path with nothing to unlearn.

The tension is that these bets optimize hard for WFL's aesthetic and beginner-friendliness while diverging from the entire body of practice — and the entire third-party tooling and supply-chain-security ecosystem — that a "Cargo/npm-grade" package manager depends on. Four expert reviewers, grounded by a researcher's fact-finding, were asked to stress-test the design adversarially and converge on what to keep, rebuild, and add.

## Research Summary

The following facts, established by the panel's researcher (Zara, July 2026), anchor every judgment below. Where an expert opinion contradicts these, the fact wins.

**Manifests and tooling.** A survey of ~60 package managers (Nesbitt, Jan 2026) found **no mainstream ecosystem uses a prose/English manifest**; all use structured formats (TOML/JSON/YAML/custom-structured). Natural language is inherently ambiguous. A prose manifest starts with **zero third-party tooling** — Dependabot, Renovate, Snyk, SBOM generators (syft), and deps.dev all parse structured formats.

**Versioning.** **CalVer carries no compatibility signal.** SemVer's major/minor/patch lets a resolver auto-decide interchangeability; "26.1 or newer" means only "released later." CalVer ecosystems (Ubuntu, pip-the-tool, Django) are apps/platforms, not consumed-as-dependency libraries, and cope via prose CHANGELOGs. **No ecosystem runs single-version selection over CalVer in production — zero precedent.**

**Resolution.** Go modules use **Minimal Version Selection (MVS)**: single-version, one operator (`>=`), no backtracking, reproducible — but only because Go **mandates intra-major backward compatibility** and pushes breaks into the import path (`mod/v2` is a different module). **PubGrub** (Dart pub, Poetry, uv, Hex, Swift PM) is the reference for human-readable conflict errors, **but** pubgrub-in-cargo is **stalled** (not renewed 2025H2; Cargo still ships its legacy resolver), and **pubgrub-rs is pre-1.0 with breaking API churn every minor release** (v0.4.0 Apr 2025, MSRV Rust 1.92) — production-proven inside uv but not API-stable for third-party embedders.

**Supply-chain state of the art.** A new registry should adopt Sigstore keyless signing (Fulcio/Rekor) + transparency log; **OIDC Trusted Publishing** (short-lived, workflow-scoped CI tokens; PyPI has >1M files via this); SLSA/PEP-740 attestations; TUF; immutable versions. **Critical caveat: signing/attestation is necessary-not-sufficient.** Miasma/Red Hat (June 2026) and TanStack (May 2026 — 84 npm artifacts, the first attack shipping **valid SLSA L3 attestations**, OIDC token lifted from a GitHub Actions runner via a "Pwn Request") both shipped malware with valid provenance. **Provenance proves where a build ran, not that it was honest.** npm's response is a **live 2FA publish gate + staged publishing** (GA May 22, 2026), which gates publication but not the post-approval circulation window. The TrapDoor campaign (34 packages across npm+PyPI+Crates.io) was caught by **cross-registry behavioral detection**, with no CVE at attack time.

**No-scripts and where risk moves.** "No install scripts" (JSR's founding premise) genuinely eliminates the postinstall attack class (Shai-Hulud worm, Sept 2025), but risk **moves** to typosquatting (industrial-scale in 2026), dependency confusion, malicious source at build/runtime, and maintainer/CI-account compromise.

**Namespaces.** Dependency confusion works because a resolver picks the higher version regardless of source; the **#1 mitigation is scoped namespaces (`@org/name`)**. WFL's **flat namespace has no structural defense.**

**Closest precedents.** **JSR (jsr.io, Deno)** is WFL's philosophical twin — from-scratch, no install scripts, immutable versions, auto Sigstore provenance — but it **kept SemVer and a structured manifest**. Deno's original pure-URL/content-addressed model **failed on both dedup (no semver) and trust ("least reliable host bricks CI")**, so Deno added JSR (central, semver-aware, immutable) while keeping HTTP imports as a "smart redirect." **Unison's content-addressed model** (SHA3-512 of the syntax tree) makes the diamond-dependency problem "simply not exist," but Unison is niche.

**AI-agent threat (new).** LLM coding agents hallucinate **~19.7% of recommended package names** (USENIX 2025, 576k samples; 51% fabrication, 38% conflation, 13% typo); **43% of hallucinations recur identically** across runs; "slopsquatting" registers those names. Attackers now optimize READMEs for LLM consumption (PromptMink "LLMO abuse"). A CISA/Five Eyes advisory (May 2026) warns that persuasive package/tool descriptions get chosen more often by agents.

**Capability enforcement.** WASI (capability-based) + the WebAssembly Component Model (WIT-declared, host-granted) is the model for turning declared `needs` into **enforced** capabilities; Deno enforces permissions at runtime (process granularity). **No registry currently ties enforced runtime capabilities to the manifest in production — this is unshipped (opportunity + research risk).**

**Windows CAFS.** pnpm's content-addressable store dedups by content hash via reflink/hardlink, **but on Windows reflinks need ReFS/Dev Drive, fall back to hardlinks on NTFS, and symlinks fall back to junctions if Developer Mode is off. WFL is Windows-first** — a real constraint.

## Expert Perspectives

**Neil Hargrove (Pragmatist).** Frames the prose manifest as "the expensive mistake" — a bespoke parser maintained forever, with zero third-party tooling. His fix is a **structured canonical manifest and lockfile**, with natural language relocated to the **tooling layer**: `wfl add` writes the structured file, `wfl why` narrates the graph in English. Kill CalVer-as-contract; use **SemVer for resolution** with CalVer as display metadata. On resolution, he **initially favored PubGrub-quality tooling but, once the pubgrub-rs churn facts landed, argued for a hand-rolled Go-style MVS** (with the import-path major-break rule) as more operationally sound than embedding a pre-1.0 crate — "a few hundred lines we control" — with Elm-style error presentation layered on top. Keep no-scripts, permissions (but **enforce** at runtime), checksummed lockfile, yank, immutability. Add day one: scoped namespaces by default, Sigstore + OIDC Trusted Publishing + transparency log + a checksum database (sumdb). Anti-slopsquat: intercept hallucinated names in the tooling layer (registry reserves names, blocks confusables, verifies existence+signature at `wfl add` time). Defer CAFS and two-person publish to v2; explicitly ranks the security wishlist — sandbox and scopes first.

**Dr. Sylvia Okafor (Compliance Architect).** Works from a full threat model (assets, trust boundaries, adversaries). Calls **no-scripts the best structural decision** and the **flat namespace the top structural concern** → scoped namespaces. Argues **permissions are security theater unless the WFL interpreter sandboxes and enforces at runtime** (deny-by-default, transitive capabilities surfaced at resolve time; WASI/Component-Model model) — and flags candidly that manifest-tied enforcement is **unshipped anywhere**, a research risk to be mitigated by shipping interpreter enforcement first. Views the natural-language manifest as a **parser-differential and homoglyph/confusable attack surface** → a structured canonical form with **one canonical, fuzzed, Unicode-normalized parser**. Day-one must-adds: transparency log + client-side sumdb verification (verifiable **without trusting the registry**, with a named trust root and key-rotation), scoped namespaces + verified publishers, OIDC Trusted Publishing. Insists **attestation is a floor not a ceiling** (TanStack/Miasma) → add live 2FA/staged publish + cross-registry behavioral detection. Treats slopsquatting as a new threat class: name reservation, a no-typosquat window, verified publishers surfaced in resolution, **agent-facing signed metadata** (agents consume a signed manifest, not a free-text README), and a resolver that returns "no such verified package" rather than a helpful near-match. Vendor the resolver behind a **trait boundary** so churn can't cascade into trust decisions.

**Marcus Venn (Frontier Engineer).** Conceded that a **prose-primary manifest is a "tooling suicide pact"** → structured canonical (TOML/JSON), prose as a lossy display/edit view. **Honestly retracted "embed pubgrub-rs"** after the stalled/churn facts → MVS-style single-version selection with PubGrub-quality, Elm-voiced error **presentation** as WFL's own layer. His signature move: **content-addressing at the core** (Unison-style — identity = hash of normalized source AST, not name+version), which fits because WFL files *are* the artifact; names/scopes become a human/AI index over hashes, and slopsquatting is blunted because a hallucinated name resolves to nothing and a pinned hash can't be swapped. Supply chain day one: immutable + Sigstore + transparency log + OIDC Trusted Publishing + SLSA/PEP-740, **plus per-release human approval and a runtime-enforced capability sandbox** (WASI Component Model) as the post-signing control that catches what signing structurally cannot. CAFS: a capability-detecting store (reflink→hardlink→copy), validated on NTFS-without-Dev-Mode first. Keep no-scripts.

**Priya Ashworth (Systems Disruptor).** Reframes the whole problem: because **WFL files are the artifact** (pure source, no build step), WFL **lacks Cargo's hardest problem** — reconstructing a build graph. WFL's real problem is **trustworthy composition of small, immutable, pure-source, English-readable modules consumed by humans and AI agents.** Proposes the manifest be a **WFL-native typed data literal as the canonical form** — reusing the compiler's own parser (no bespoke grammar, no TOML) so the English aesthetic is kept *honestly* — with prose as a rendered view. SemVer with **Go-style enforced intra-major backward-compat** (import-path rule) to make single-version selection *sound*; CalVer as metadata; implement MVS rather than embed pre-1.0 pubgrub-rs. **Withdrew "registry as index, not gatekeeper"** after the Deno-failure fact → corrected to a **central authoritative index + availability layer (JSR-shaped)** for discovery/naming plus content-addressing for verification, so no client must trust the registry's integrity (client-side sumdb + transparency log). States a **two-layer trust doctrine**: (1) **integrity** = content-addressing + sumdb + transparency log (cheap, total); (2) **honesty** = which signing *cannot* provide, requiring two-person review, per-release approval, and behavioral/anomaly signal. Pushes **AI-agent-first affordances**: scopes-only, name reservation + verified-publisher binding + published reservation list, a `wfl why <dep>` that surfaces first-seen date / verification / whether a name was recently reserved due to hallucination reports, and an **agent-facing signed capability manifest** (the agent reads a signed contract, not an attacker-written README). Capability enforcement via WASI Component Model is a **roadmap item (research risk), not a day-one blocker.**

**Good-faith-update highlight:** two of the four experts — **Marcus and Neil** — explicitly **retracted their initial "embed pubgrub-rs" position** once the researcher surfaced the stall and pre-1.0 API churn. Convergence *after* a reversal on stated facts is the strongest possible signal in an adversarial review.

## Key Trade-offs

Several genuine tensions survived peer review. These are real forks, not artifacts of miscommunication, and the panel did not paper over them.

**1. Canonical manifest format: TOML/JSON vs WFL-native typed literal.**
Neil and Marcus (as his fallback) favor a **boring, structured TOML/JSON** canonical form — the maximally interoperable choice, the one every third-party tool (Dependabot, Renovate, Snyk, syft, deps.dev) already parses. Priya counters that a **WFL-native typed data literal reusing the compiler's own parser** avoids a bespoke grammar *and* a foreign format, preserves the English aesthetic honestly, and means there is only ever one parser to fuzz. The trade is **ecosystem interoperability and off-the-shelf tooling (TOML/JSON) vs aesthetic coherence and parser unification (WFL literal)**. Both satisfy the No-Unlearning Invariant equally; both are structured. Sylvia's security lens is neutral between them provided the parser is single, fuzzed, and Unicode-normalized — but she'd note a WFL-literal manifest inherits the language's full parser as attack surface, whereas TOML has a small, hardened, well-understood grammar. **This is the one decision the panel could not close.**

**2. Identity model: content-addressing at the core vs name+version+MVS.**
Marcus and Priya want **content-addressed identity** (hash of the normalized source AST) as the primitive, with names/scopes as a human/AI index — which structurally kills version-swap and blunts slopsquatting, and fits WFL because files *are* the artifact (Unison precedent). Neil prefers the **simpler name+version+MVS base** he can ship and reason about, treating content hashes as lockfile checksums rather than the identity primitive. The trade is **structural trust elegance and future-proofing (content-addressed) vs operational simplicity and precedent (name+version)**. The research cuts both ways: Unison proves content-addressing solves diamond dependencies, but Unison is niche and Deno's pure-content-addressed model *failed* on dedup and trust until it added a central semver-aware index — which is why even the content-addressing advocates (Priya explicitly) want a **central authoritative index layered over** content hashes, not content-addressing alone.

**3. How much AI-agent-first design to commit day one.**
Priya and Sylvia want agent-facing signed metadata, name-reservation lists, and hallucination-aware `wfl why` **from the start**, arguing the 19.7% hallucination rate and 43% recurrence make this a present, not future, threat. Neil agrees on the *mechanism* (intercept hallucinated names at `wfl add` time) but scopes it more tightly to name reservation + confusable-blocking + existence/signature verification, deferring the richer agent-metadata surface. The trade is **comprehensive agent-first affordances now vs a minimal anti-slopsquat core now, richer agent UX later**.

**4. Runtime capability enforcement: genuine value vs unshipped-anywhere research risk.**
All four agree declared-only permissions are theater and enforcement is the goal. But the research is blunt: **no registry ties enforced runtime capabilities to the manifest in production.** Marcus and Sylvia want interpreter enforcement shipped early (Sylvia: "ship interpreter enforcement first" to de-risk); Priya labels full WASI Component Model enforcement a **roadmap item, not a day-one blocker**. The trade is **shipping the differentiating security control early (and absorbing genuine research risk) vs shipping declared permissions now and enforcing incrementally**. The panel's center: enforce at the interpreter level as fast as is safe, but do not let the unshipped full-capability model block the registry launch.

## Ideas Worth Highlighting

**Priya's reframe — WFL doesn't have Cargo's hardest problem.** Because WFL packages are pure source with no build step, the manifest *is* the artifact and there is no build graph to reconstruct. This inverts the usual framing: WFL's job is not "reproduce a compilation" but **"trustworthy composition of small, immutable, English-readable modules consumed by humans *and* AI agents."** This reframe is why content-addressing and agent-first design are natural fits rather than bolt-ons, and it should headline the rebuild's design rationale.

**Marcus's content-addressed identity.** Making identity the hash of the normalized source AST (Unison's insight, adapted) means a pinned dependency literally cannot be swapped, a hallucinated name resolves to nothing, and diamond dependencies stop existing. Even if the panel doesn't adopt it as *the* primitive on day one (see trade-off 2), content hashes as the lockfile trust anchor are non-negotiable, and the AST-hash idea is the strongest available structural answer to slopsquatting and version-swap attacks.

**"English as a tooling/view layer, structured as source of truth" — the No-Unlearning reconciliation.** The single most important synthesis of the review: the natural-language aesthetic that motivated the prose manifest is **fully preservable without prose being the source of truth.** `wfl add http-client` writes the canonical structured file; `wfl why http-client` renders the graph in English; the manifest can be *displayed* and *edited* as English while being *stored and parsed* as structured data. This satisfies the No-Unlearning Invariant — the beginner reads and writes English, the expert reads and writes the same English, and nothing learned must be unlearned — while restoring the entire third-party tooling ecosystem and eliminating the ambiguity, parser-differential, and homoglyph attack surface of a prose canonical form. The beginner form and expert form remain the same form; only the *storage* format changes, invisibly. This is the reconciliation that lets WFL keep its soul and its safety.

## Risks and Unknowns

- **pubgrub-rs churn.** Pre-1.0, breaking every minor release, not API-stable for embedders. Mitigation is to *not* embed it and implement MVS behind a trait boundary (Sylvia's point) so a future resolver swap cannot cascade into trust decisions.
- **MVS-over-CalVer has zero precedent.** No production ecosystem runs single-version selection over CalVer. Adopting SemVer + Go-style enforced intra-major compat + import-path major breaks is what makes single-version *sound* — but WFL would still be pioneering the combination, and the Go-style compat guarantee must be *enforced* (by tooling/CI), not merely requested, or single-version selection silently breaks consumers.
- **Manifest-tied capability enforcement is unshipped anywhere.** The most differentiating security feature is also the least precedented. Genuine research risk; de-risk by shipping interpreter-level enforcement first and treating full WASI Component Model integration as a roadmap milestone.
- **Windows CAFS is genuinely constrained.** Reflinks need ReFS/Dev Drive; NTFS falls back to hardlinks; symlinks fall back to junctions without Developer Mode. WFL is Windows-first, so any content-addressable store must be a capability-detecting store (reflink→hardlink→copy) and must be validated on **NTFS without Developer Mode first**, not last.
- **Signing is not sufficient.** TanStack shipped valid SLSA L3 attestations; Miasma shipped with valid provenance. Provenance proves *where*, not *honest*. Any security roadmap that stops at Sigstore/OIDC/attestation is incomplete by demonstrated fact; per-release human approval and cross-registry behavioral detection are required, not optional.
- **Registry longevity and governance.** wflhub.org is a central authoritative index the whole ecosystem would depend on. Deno's history shows the decentralized alternative *failed*, so centrality is correct — but it concentrates operational, funding, and governance risk. The transparency log + client-side sumdb design partly mitigates this (clients don't have to *trust* the registry's integrity), but availability, key custody/rotation, and a named trust root remain open governance questions.
- **AI-agent threat is moving fast.** The 19.7% hallucination rate and README-for-LLM optimization are 2025–2026 findings; the attacker playbook is actively evolving, so agent-facing defenses must be treated as a living surface, not a one-time feature.

## Recommendation

The panel's center of gravity is unambiguous: **keep wflpkg's security instincts, rebuild its three core mechanisms, and add the modern supply-chain and agent-defense stack.** The existing ~6,000 lines encode good instincts (no-scripts, permissions, checksums, yank, immutability, a from-scratch registry) wrapped around three mechanisms that the research shows are unsound or unsupportable. Rebuild the three; keep and harden the rest.

**Recommended concrete stack:** structured canonical manifest + lockfile with an English `wfl add`/`wfl why` tooling layer; SemVer + enforced intra-major backward-compat + import-path major breaks; MVS resolver behind a trait boundary with Elm-voiced conflict presentation; scoped namespaces only; content hashes as the lockfile trust anchor; Sigstore + OIDC Trusted Publishing + transparency log + client-side sumdb with a named trust root and key rotation; per-release human approval + 2FA/staged publish + behavioral detection; interpreter-enforced capabilities now, WASI Component Model on the roadmap; agent-facing signed package metadata and a published name-reservation list; no install scripts; CAFS deferred to v2.

**The one real open decision to put back to Brad.** The panel closed everything except two intertwined questions, and they should be decided by the owner because they trade WFL's aesthetic against ecosystem interoperability:

1. **Canonical manifest format — TOML/JSON vs a WFL-native typed data literal reusing the compiler's own parser?** TOML/JSON buys instant compatibility with every existing supply-chain tool (Dependabot, Renovate, Snyk, syft, deps.dev) and a small hardened grammar; the WFL-typed literal buys aesthetic coherence and a single parser to fuzz, at the cost of exposing more parser surface and forfeiting off-the-shelf tooling. Both are structured; both satisfy No-Unlearning; the English *view* is identical either way.
2. **Content-addressed identity — yes or no?** Adopt hash-of-normalized-AST as the identity primitive (Marcus/Priya — maximal structural trust, kills version-swap and slopsquatting, but novel and, per Deno, only viable *with* a central semver-aware index over it), or keep name+version+MVS as the base with content hashes used as lockfile checksums only (Neil — simpler, precedented, shippable).

Everything else is settled. These two are genuine values-and-risk calls, not technical unknowns, and they belong to the owner.

## Next Steps

1. **Decide the two open questions first** (manifest canonical format; content-addressed identity yes/no). They gate the data model and cannot be deferred without churn.
2. **Write the structured canonical schema and the English view layer.** Specify the canonical manifest/lockfile, then implement `wfl add`/`wfl why` as the English read/write surface. Build one canonical parser — fuzzed and Unicode-normalized — and nothing else that parses manifests.
3. **Define and *enforce* the versioning contract.** SemVer + Go-style intra-major backward-compat + import-path major breaks, with CalVer relegated to display metadata. Ship the compat check as tooling/CI so single-version selection stays sound; without enforcement this bet fails silently.
4. **Implement the MVS resolver behind a trait boundary**, with Elm-voiced conflict presentation as WFL's own layer. Do not embed pubgrub-rs. The trait boundary is a security requirement (Sylvia): resolver churn must never touch trust decisions.
5. **Stand up the registry trust core:** scoped namespaces, immutable versions, Sigstore + OIDC Trusted Publishing + transparency log + client-side sumdb with a named trust root and key-rotation policy. Publish the name-reservation list and verified-publisher bindings.
6. **Ship interpreter-level capability enforcement early** (deny-by-default, transitive capabilities surfaced at resolve time). Treat full WASI Component Model integration as a tracked roadmap milestone with an explicit research spike.
7. **Layer the honesty controls** signing cannot provide: per-release human approval, live 2FA/staged publish, and behavioral/anomaly detection (ideally cross-registry-aware).
8. **Land the agent-first anti-slopsquatting surface:** resolver returns "no such verified package" (never a helpful near-match), confusable/homoglyph blocking, and agent-facing *signed* metadata so agents consume a signed contract rather than an attacker-authored README.
9. **Validate, then defer, CAFS.** Prototype the capability-detecting store on **NTFS without Developer Mode first**; only then extend to ReFS/Dev Drive reflinks. Do not let CAFS block launch.
10. **Prove the spine before scaling.** Publish a handful of real first-party packages through the full pipeline (structured manifest → MVS resolve → signed + logged publish → client-side verification → enforced capabilities) and confirm third-party tooling can parse the canonical format, before opening the registry to the public.
